Design specification

The SnsDrv sensor

A kernel driver that watches the OS event surface, ships normalized telemetry to a userland engine, and enforces the engine's block decisions inline — routing latency onto the one path that actually needs it.

01
Overview

Three planes, one port

Every design choice below serves a single goal: enforcement latency falls only on events that can actually block. The bulk of traffic is fire-and-forget telemetry; a tiny, dynamically-chosen set of operations is enforced synchronously; and a control sideband tells the kernel which ones.

Plane 01

Telemetry

Every observed event, shipped up asynchronously so the engine can build its forensic storyline. Blocks nothing; never waits.

dir   kernel → service send  FltSendMessage · no reply path  callback → Queue → Worker
Plane 02

Enforcement

The rare armed / chokepoint event. Sent synchronously, the callback blocks for a verdict, then allows or denies the operation in place.

dir   kernel ⇄ service send  SendWithReply · 20 ms · fail-open path  callback → EnforceSync
Plane 03

Control

The engine pushes ARM / DISARM down so the kernel knows which (pid, op) to intercept. Also registers the service pid for exemption.

dir   service → kernel send  FilterSendMessage · 16 B recs path  MessageNotify → ArmTable
02
Data flow

Architecture

Kernel event sources on the left feed a routing decision. Non-armed events take the async telemetry lane through the queue and worker thread; the armed identity and the static lsass chokepoint take the sync enforcement lane straight out. The control lane runs back the other way, keeping the arm table current.

Telemetry (async) Enforcement (sync + verdict) Control (arm pushdown)

The queue is not batching — it is the IRQL bridge. FltSendMessage requires PASSIVE_LEVEL; thread-notify runs at APC and WFP at DISPATCH, so those callbacks must hand off to the PASSIVE worker thread. Enforcement, only ever raised from PASSIVE callbacks, sends directly.

04
Reference

Components

Entry.cpp

Driver owner

DriverEntry builds the Driver object that owns and wires every subsystem, exposes globals for the callbacks, and tears down in reverse.

  • Enforcement thunk callback → Worker::EnforceSync
  • Strict init / cleanup ordering (see §07)
MiniFilter.cpp

Filter · Port · Connection telemetry

Registers the minifilter (IRP_MJ_CREATE → FileOpen), owns the comms port, and wraps the client connection.

  • Send — fire-and-forget telemetry
  • SendWithReply — sync verdict, 20 ms
  • ControlMessageNotify — arm pushdown sink
ProcessMonitor.cpp

Process monitor enforce

ObCallbacks + process/thread notify routines. The enforcement decisions live here.

  • lsass handle-open → strip sensitive access
  • armed spawn → STATUS_ACCESS_DENIED
  • exempts the service pid (self-deadlock guard)
Worker.cpp

Queue · Worker bridge

System thread that drains the queue and sends each event immediately (no batching), and publishes the live connection for the enforcement path.

  • IRQL bridge for APC / DISPATCH callbacks
  • Publishes connection under a lock, unpublished before teardown
ArmTable.cpp

Arm table control

Small lock-guarded set of armed identities. Consulted on the hot path; updated by the control channel.

  • Key (pid, start_ms, op) · shared-lock lookup
  • IsArmed / Arm / Disarm / SetServicePid
Network.cpp

WFP monitor scaffold

Registers WFP callouts and extracts IP / port / pid, but does not yet emit events. Kept as scaffolding for future network telemetry (engine already has Op::Connect).

05
Protocol

Wire format v2

Records are engine-ready: numeric fields sit at fixed, naturally-aligned offsets, every record carries the full process identity (pid, create_time) inline, and each record is size-prefixed and padded to 8 bytes so the service reads fields directly with no cursor parse and can skip unknown types.

Identity

start_ms = FILETIME / 10000 — the same engine milliseconds the service uses. The driver matches an arm by pid plus its own creation FILETIME truncated to ms, so a reused pid never inherits a stale arm (collision only within the same millisecond).

06
Mechanism

Arm vs. block

Two different things at two different moments. Arm is preparation; block is the act. Arm exists so that, when the awaited operation finally arrives, the block decision can be made synchronously and in time to stop it.

◇ ARM

“Next time this process does this op, be ready to intercept.” Pushed down before the dangerous action, it re-routes a future event onto the sync path. Blocks nothing by itself.

when
score ≥ θ_block, chokepoint not yet reached
state
ArmTable entry (pid, start, op)
effect
routing: async → sync enforcement
■ BLOCK

The decision to deny an operation happening right now. At the driver it becomes a concrete action: strip the handle's access bits, or fail the process creation.

when
the armed event arrives · at the chokepoint
state
verdict = Deny
effect
strip access · STATUS_ACCESS_DENIED
// ransomware storyline
exec powershell → inhibit recovery → [score > θ_block]  ◇ ARM (pid, write)
  … await …
first encrypting write arrives → armed → sync send → ■ BLOCK — deny the write

Single-step chokepoints — like the lsass read — cannot be pre-armed: the first matching event both crosses the threshold and is the chokepoint, so it blocks directly via the static predicate. Arming only helps multi-step patterns.

07
Ownership

Lifecycle

The Driver object constructs subsystems in dependency order and tears them down so that nothing referencing the arm table or the connection outlives it.

Construct →
  1. ArmTable must outlive all consumers
  2. Queue event staging
  3. Worker + arm table ref
  4. MiniFilter::Filter start filtering
  5. Port \SnsDrvPort + control sink
  6. ProcessMonitor arms + enforce thunk
  7. Network WFP callouts
← Destruct
  1. Port stop new connections + control
  2. ProcessMonitor stop enforcing callbacks
  3. Network unregister callouts
  4. Worker stop thread, drop connection
  5. Filter stop new events
  6. Queue drain
  7. ArmTable last — no referrers remain
