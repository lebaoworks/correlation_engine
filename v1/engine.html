Vấn đề

Nhận diện một cuộc tấn công cần nhiều event theo thời gian. Nhưng để chặn, kernel phải trả lời allow/deny đồng bộ trong micro-giây — không kịp chờ ghép cả chuỗi.

Cách giải

Duy trì trạng thái chuỗi ở userland. Khi đủ tin cậy → vũ trang (arm) kernel để nó tự deny hành động enforcing kế tiếp (điểm nghẽn: mã hoá / đọc LSASS / exfil). Chặn tại chỗ, không hồi tố.

Kiến trúc

Kernel là cảm biến & tay chặn. Userland là bộ não.

Một event: sensor kernel → đẩy lên bộ não → chấm điểm → verdict/arm đi xuống → kernel thực thi. Correlation graph không nằm trong kernel.

Chống false positive · Variable binding

"Ghi X rồi chạy X" ≠ "ghi X rồi chạy Y"

Ghi X → chạy đúng X
write · bind dropped = F_x
exec · image = F_x
F_x == F_x ✓
SUSPECT · ALLOW  installer/updater làm y hệt → điểm thấp, không chặn.
Ghi X → chạy Y khác
write · bind dropped = F_x
exec · image = F_y
F_y ≠ F_x ✗
BINDING REJECT  mẫu dropper không khớp → không FP.

Binding so khớp theo file identity (FileId), không theo path → rename không né được, "chạy file khác" không dính oan.

Bốn đảm bảo

◆O(1) / event — bitmask + DSU + hash, không subgraph matching NP-hard trên hot path.
◆Chặn tại điểm nghẽn — arm kernel, deny hành động enforcing kế tiếp, không hồi tố.
◆Thứ tự tự do — partial-order DAG; thứ tự là điểm thưởng, không phải điều kiện.
◆FP có kiểm soát — binding theo identity; ngưỡng bảo thủ; audit-only trước khi enforce.
